3SXW HARD FORK GUIDE

Purpose
This document is a working guide for future chats and contributors before making any change to this repository.
It exists to preserve the real intent of this hard fork and to prevent unnecessary refactors, wrong assumptions, or changes that fight the project's direction.

Project Identity
3SXW is a hard fork of 3SX, itself based on a decompilation of the PlayStation 2 port of Street Fighter III: 3rd Strike.
The goal of this fork is not strict 1:1 code purity with the original PS2 codebase.
The goal is to keep the game faithful enough as a modern desktop port while adding practical improvements that make the project easier to use, easier to distribute, and more complete for local play.

This fork intentionally accepts:
- modern platform adaptations;
- desktop quality-of-life features;
- targeted gameplay or usability adjustments;
- reverse-engineered code that is not architecturally perfect, as long as it works correctly for the fork's goals.

This fork does not prioritize:
- broad refactors for code elegance alone;
- removing desktop-specific conveniences;
- cleaning reverse-engineering artifacts just because they are imperfect;
- online-focused development;
- strict preservation of upstream behavior when this fork has already chosen a different direction.

Core Intent Of This Fork
The main goals of this hard fork are:
- fully functional save support, matching the expected console experience in practice;
- fully functional replay support, matching the expected console experience in practice;
- Gill available from the start, without requiring long unlock grinding;
- all generated files kept local to the game/build folder;
- easier usage on modern hardware;
- a practical and portable local/offline experience.

Decisions Already Made And Intentionally Preserved
Do not treat the following as accidental deviations. These are intentional fork decisions.

1. Akuma gameplay fixup stays
The Akuma adjustment in src/port/char_data.c is intentional and should be preserved unless the user explicitly asks to remove or rework it.

2. Gill unlocked from the start stays
Gill being available from the start is intentional and should remain.
This fork does not currently plan to support online play as a priority, so online balance is not a guiding requirement.

3. Local file layout stays
Saves, config, keymap, and other generated files staying inside the project/build folder is intentional.
Do not move them back to OS-specific roaming/appdata-style locations.

4. Desktop configuration options stay
Options such as fullscreen, window sizing, scale modes, and visual presentation controls are intentional and should remain.

5. Desktop convenience features stay
Features such as screenshot support, fullscreen hotkeys, windowed operation, and other desktop usability improvements are intentional.

6. Local asset extraction stays
Extracting required assets from an official PS2 ISO into local files is intentional and should remain.
The project targets modern hardware and local execution, not direct media access patterns from the original console environment.

7. Portability layers stay
SDL-based platform code, local path handling, modern input handling, adapted audio, and related platform layers are core to the fork and should not be treated as temporary hacks to remove.

8. Structural differences from the original codebase are acceptable
This repository comes from reverse-engineering work.
Not all code will match the original structure or style exactly.
If something works and supports the fork's goals, do not propose refactoring it just because it is not elegant.

What Future Chats Must Not Assume
- Do not assume the project wants to become a strict PS2 1:1 restoration.
- Do not assume all gameplay-affecting differences must be removed.
- Do not assume desktop-specific behavior is unwanted.
- Do not assume architectural cleanup is automatically valuable.
- Do not assume unused online code must be removed immediately.
- Do not assume reverse-engineered rough edges are bugs unless they cause real functional problems.

Current Position On Netplay
Netplay exists in the codebase, but it is not part of the current priority for this fork.
The current decision is:
- do not invest in online work unless explicitly requested;
- do not remove netplay code preemptively if it is not causing problems;
- do not let online-related design concerns override local/offline goals.

Current Position On Tools And Extra Files
This fork should keep only what is actually useful.
Development support files may be removed if they are redundant and not needed for build or runtime.
However, future chats must verify whether a file is referenced by docs, workflows, build scripts, or runtime before recommending removal.

Implementation Philosophy
When making changes, prefer this order of priorities:

1. Preserve working behavior that matches the fork's intent.
2. Improve local usability and portability.
3. Improve save and replay reliability.
4. Make targeted fixes with low blast radius.
5. Avoid broad refactors unless explicitly requested.

In practice, prefer:
- minimal, surgical changes;
- preserving established user-facing behavior;
- improving implementation without rewriting subsystems;
- solving the real functional problem instead of polishing unrelated code.

Areas That Deserve Extra Care
Changes in these areas should be analyzed carefully because they affect the fork's core goals:
- src/port/sdk/ and memory-card/save flow;
- replay-related code paths;
- src/port/paths.c and local file organization;
- src/port/resources.c and asset extraction flow;
- src/port/config/ and input/config persistence;
- src/port/char_data.c and gameplay-related fork adjustments;
- rendering and audio paths when they affect real gameplay behavior or usability.

Areas That Are Lower Priority Unless Explicitly Requested
- netplay and matchmaking;
- broad cleanup of reverse-engineered naming/style;
- replacing working systems just to make the architecture cleaner;
- removing convenience features that improve desktop use.

Rules For Future Chats
Before any code change:
- analyze the request in the context of this fork's intent;
- state which files will be changed before editing;
- do not edit files until the user has requested a change;
- do not run builds unless the user explicitly changes the workflow expectation.

Repository workflow expectations from the project owner:
- do not build the project automatically;
- the user performs builds locally;
- always provide an analysis of the request first;
- always inform the user which files will be changed.

How To Evaluate A Proposed Change
A change is likely aligned with this fork if it:
- improves local play;
- improves portability;
- improves save or replay behavior;
- improves usability on modern hardware;
- fixes a real functional problem;
- preserves established intentional fork behavior.

A change is likely misaligned if it:
- removes an intentional fork feature;
- pushes the project toward strict upstream purity without user request;
- requires large refactors without strong payoff;
- prioritizes online infrastructure over local goals;
- complicates distribution or local file organization.

Short Summary
3SXW is a practical hard fork for modern local play, not a purity restoration project.
Keep the fork-specific improvements.
Prefer targeted fixes over refactors.
Preserve portability, save/replay goals, and desktop usability.
Treat existing intentional differences as part of the product unless the user explicitly decides otherwise.
